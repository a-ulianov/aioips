# Обработка ошибок и повторы

Это руководство — про то, как `aioips` сообщает об ошибках и как писать устойчивый код: какая
иерархия исключений, что значит каждый HTTP-код, какие ошибки клиент повторяет сам, а какие надо
обрабатывать вам.

!!! tip "Главная идея"
    Все ошибки клиента — это подклассы одного исключения `IPSError`. Ловите `IPSError`, чтобы
    перехватить любую проблему, или конкретный подкласс, чтобы реагировать на конкретную причину.
    Транзиентные ошибки (сеть, 429, 5xx) клиент **повторяет автоматически** — отдельно их обрабатывать
    обычно не нужно.

## Иерархия исключений

Все исключения импортируются из `aioips`:

```python
from aioips import (
    IPSError,                 # корень иерархии
    IPSClientError,           # 400
    IPSAuthError,             # 401
    IPSForbiddenError,        # 403
    IPSNotFoundError,         # 404
    IPSConflictError,         # 409
    IPSTooManyRequestsError,  # 429
    IPSServerError,           # 5xx
    IPSConnectionError,       # сеть / таймаут (status == 0)
)
```

Дерево:

```text
IPSError                     (status, message, title, detail, payload)
├── IPSClientError           400  — некорректный запрос
├── IPSAuthError             401  — не аутентифицирован / истёк токен
├── IPSForbiddenError        403  — нет прав
├── IPSNotFoundError         404  — ресурс не найден
├── IPSConflictError         409  — конфликт состояния (ЖЦ/блокировка)
├── IPSTooManyRequestsError  429  — превышен лимит запросов
├── IPSServerError           5xx  — внутренняя ошибка сервера
└── IPSConnectionError       —    — сеть/таймаут (HTTP-ответа нет, status == 0)
```

Каждое исключение несёт поля для диагностики: `status` (HTTP-код или `0`), `message`, `title` и
`detail` (из тела ошибки IPS формата `ApiProblemDetails`), `payload` (полное тело ответа).

## Что значит каждый код

| Класс | Код | Причина | Повторяется? | Что делать |
|---|---|---|---|---|
| `IPSClientError` | 400 | Невалидное тело/параметры запроса | Нет | Исправьте входные данные |
| `IPSAuthError` | 401 | Нет токена / истёк / неуспешный вход | Авто-refresh 1 раз | Проверьте креды; см. ниже |
| `IPSForbiddenError` | 403 | Аутентификация есть, прав нет | Нет | Смените роль/уровень доступа |
| `IPSNotFoundError` | 404 | Ресурс не найден | Нет | Часто — передан id версии вместо id объекта |
| `IPSConflictError` | 409 | Конфликт ЖЦ / блокировка | Нет | Сделайте checkout, проверьте шаг ЖЦ |
| `IPSTooManyRequestsError` | 429 | Превышен лимит запросов | **Да** | Обычно ничего — клиент повторит |
| `IPSServerError` | ≥500 | Внутренняя ошибка сервера | **Да** | Клиент повторит; если не помогло — на сторону сервера |
| `IPSConnectionError` | 0 | Сеть/таймаут | **Да** | Клиент повторит; проверьте доступность сервера |

### Подробности по частым кодам

!!! warning "404 — почти всегда путаница id объекта и версии"
    Самая частая причина 404 (или `None` у методов чтения) — передан **id версии** (`F_ID`) туда, где
    ждут **id объекта** (`F_OBJECT_ID`). Сверьтесь с концепцией [Объект и версия](../concepts/data-model.md).

!!! warning "409 — нарушен жизненный цикл"
    Запись без checkout, объект занят другим пользователем, недопустимый шаг ЖЦ, атрибут `read_only` —
    всё это `IPSConflictError`. Перед записью извлекайте объект на редактирование (см.
    [Редактирование объектов](editing-objects.md)).

!!! warning "401 — клиент пытается обновить токен сам"
    При первом `401` ядро один раз обновляет JWT-токен и повторяет запрос. Исключение `IPSAuthError`
    поднимается, только если и это не помогло (неверные креды, нет способа повторного входа).

## Как ловить ошибки

### Перехватить любую ошибку клиента

```python
from aioips import IPSError

async with IPSClient(config=config) as ips:
    try:
        obj = await ips.object_get(102550)
    except IPSError as exc:
        # Доступны exc.status, exc.message, exc.title, exc.detail, exc.payload
        print(f"Ошибка IPS {exc.status}: {exc.message}")
        raise
```

### Реагировать на конкретные причины

```python
from aioips import IPSConflictError, IPSForbiddenError, IPSNotFoundError
from aioips.schemas.objects import AttributeValues

async with IPSClient(config=config) as ips:
    try:
        working_id = await ips.object_check_out(102550)
        await ips.object_set_attribute_values(
            working_id,
            [AttributeValues(attribute_id=9, values=["550.07.305"])],
        )
        await ips.object_check_in(working_id)
    except IPSConflictError:
        print("Объект занят или шаг ЖЦ не позволяет правку")
    except IPSForbiddenError:
        print("Недостаточно прав на запись")
    except IPSNotFoundError:
        print("Объект не найден — проверьте, что это id объекта, а не версии")
```

## Автоматические повторы (tenacity)

Клиент сам повторяет **транзиентные** ошибки с экспоненциальной задержкой. Повторяются:

- `IPSServerError` (5xx),
- `IPSTooManyRequestsError` (429),
- `IPSConnectionError` (сеть/таймаут).

**Не повторяются** детерминированные ошибки: `400`, `401` (кроме одноразового авто-refresh токена),
`403`, `404`, `409` — их повтор бессмыслен, нужно исправлять запрос или состояние.

Поведение настраивается через `IPSConfig` (значения по умолчанию):

| Параметр | По умолчанию | Смысл |
|---|---|---|
| `max_retries` | `3` | Сколько раз повторить транзиентную ошибку |
| `retry_min_wait` | `1.0` | Минимальная задержка между повторами, сек |
| `retry_max_wait` | `10.0` | Максимальная задержка между повторами, сек |

```python
from aioips import IPSConfig

config = IPSConfig(
    max_retries=5,        # больше попыток
    retry_min_wait=0.5,
    retry_max_wait=20.0,
)
```

!!! tip "Повторы безопасны для чтения и для multipart-загрузок"
    Чтение идемпотентно, поэтому повтор не вредит. Загрузка файла (`upload_temp_file`) тоже
    retry-безопасна: содержимое полей формы пересобирается на каждой попытке.

!!! warning "Не превращайте 4xx в повторы вручную"
    `400`/`403`/`404`/`409` означают, что запрос или состояние неверны. Повторять их в цикле
    бесполезно — исправьте причину (id, права, checkout, тело запроса).

## Чек-лист устойчивого кода

1. Оборачивайте вызовы в `try/except IPSError` (или конкретные подклассы) там, где ошибка ожидаема.
2. На `404`/`None` — первым делом проверьте, что передаёте **id объекта**, а не версии.
3. На `409` при записи — убедитесь, что сделали `object_check_out` и пишете по id рабочей копии.
4. Полагайтесь на встроенные повторы для сети/429/5xx; не дублируйте их вручную.
5. Логируйте `exc.status` и `exc.detail` — они дают точную причину со стороны IPS.
