# Участие в разработке

Эта страница — стандарты разработки `aioips`: как устроен проект, какой код проходит ревью, как
именуются методы и почему каждый метод проверяется на боевом сервере. Цель — чтобы обёртка
масштабировалась на сотни эндпоинтов **без потери качества и единообразия**.

## Язык: код английский, документация русская

Жёсткое правило проекта:

- **Английский** — код, имена, коммиты, ветки, CLI-вывод (краткость и совместимость с инструментами).
- **Русский** — документация и docstrings (Google-style), README, комментарии в коде.

## Окружение и инструменты

- Python **3.13**, src-layout (`src/aioips/`), пакеты/модули в `lowercase_with_underscores`.
- Виртуальное окружение — всегда `.venv/` в корне.
- `pyproject.toml` — **единственный** источник конфигурации (setuptools); без `setup.py`, без
  `requirements.txt`. Экстры: `test`, `dev`. Менеджер пакетов: `uv` (предпочтительно) или `pip`.
- `ruff` — линт и формат (без black/isort): `line-length = 100`, `target-version = "py313"`,
  правила `E,W,F,I,N,UP,B,ASYNC,RUF,D,S` (S = bandit), конвенция docstring — `google`.
- `mypy --strict` с плагином `pydantic.mypy`; послабления только для `tests.*`.
- `pytest` (`asyncio_mode = "auto"`), покрытие с `branch = true` и порогом **80%**.

Типизация — 100% (mypy strict). Нативный синтаксис: `list[X]`, `dict[k, v]`, `X | None`. Без
`from __future__ import annotations`. `Any` избегать (и обосновывать, если неизбежен).

## Гейт «готово» (обязателен)

Изменение считается готовым, только когда **весь гейт зелёный**:

```text
ruff check .            # линт (включая bandit-правила S)
ruff format --check .   # форматирование
mypy src                # строгая типизация
pytest                  # тесты + порог покрытия (--cov-fail-under=80)
```

Если что-то красное — изменение не сделано. Чинить нужно **корневую причину**, а не симптом. Тот же
гейт прогоняется в CI (GitHub Actions) на каждый push; CI должен быть зелёным.

## Конвенция имён (ADR-0003)

Имена методов, файлов и схем **выводятся детерминированно из адреса эндпоинта** — это убирает споры об
именах и позволяет генерировать разделы. Берётся часть пути **после `/api/`** (работает и для
`/core/api/...`, и для `/api/...`), сегменты-параметры `{...}` отбрасываются.

1. **Раздел (mixin-каталог)** = первый сегмент после `/api/`: `objects`, `metadata`, `files`. Это
   директория в `methods/<раздел>/` и `schemas/<раздел>/`.
2. **Имя файла/метода** = оставшиеся непараметрические сегменты в `snake_case` через `_`:
   - `…/objects/CreateByPrototype` → `object_create_by_prototype`
   - `…/objects/{objectId}/objectInfo` → `object_info`
   - `…/objects/byGuid/{guid}/objectInfo` → `object_info_by_guid`
3. **CRUD без описательного хвоста** — глагол добавляется по HTTP-методу с опорой на `operationId`:
   `GET /objects/{id}` → `object_get` (`Objects_GetObject`); `POST /objects` → `object_create`.
4. **Схемы** — `PascalCase` того же имени + суффикс роли (`…Request`, `…Response`, `…Item`); имена
   файлов схем совпадают с именами файлов методов.
5. **Имя метода клиента глобально уникально** (все mixin'ы плоско в `IPSClient`), поэтому уже несёт
   контекст раздела: `object_*`, `attribute_*`, `relation_*`, `life_cycle_*`.

`operationId` IPS имеет вид `Category_Action` — он на английском и точен; используется как ориентир
при неоднозначном пути и как основа англоязычного резюме в docstring.

### Разворот result-обёрток

Result-обёртки IPS вида `…NullableResultDto { entity, isEntityPresent }` разворачиваются **внутри
метода**: наружу отдаётся `Schema | None`, деталь транспорта не протекает в код пользователя.

## Docstrings уровня MCP

На решении будут строиться портал взаимодействия и MCP-сервер, поэтому docstring каждого публичного
метода — это **самодостаточная спецификация инструмента**: агент, прочитав только docstring, должен
понять, что метод делает, когда применять и как вызвать. Требуемые блоки (Google-style, по-русски):

- **Summary** (одна строка) — что делает.
- Расширенное описание: назначение, **когда применять**, связь с другими методами, **предусловия**
  (например, «требует checkout», «нужен id версии, а не id объекта»).
- **Args** — для каждого параметра смысл, формат, **id-пространство** (версия `objectID` vs объект
  `ObjectID`), допустимые значения, дефолт, побочные эффекты.
- **Returns** — точная структура/тип, смысл значимых полей, семантика `None`/пустого результата.
- **Raises** — какие исключения и при каких условиях.
- **Example** — реалистичный рабочий вызов.
- **Notes/References** — ограничения, грабли, `operationId`, связанные методы.

## Гейты подтверждения на разрушающих операциях

Необратимые и массовые операции (удаление объекта/связи/снимка, очистка атрибута, отмена правок,
исключение из состава, удаление истории) принимают параметр `confirm: bool = False`. Без `confirm=True`
метод **бросает ошибку и ничего не делает** — защита от случайного запуска. Перед разрушающим
изменением делается бэкап и предусматривается откат.

## Тестирование: качество, не процент

- Тесты проверяют **поведение, контракты, краевые случаи и регрессии** — то, что реально сломалось бы
  у пользователя. Тупой тест хуже его отсутствия; покрытие (≥80% общий, критические пути ~100%) —
  сигнал здоровья, а не цель.
- Предпочтение — поведенческим/контрактным тестам через публичную поверхность клиента; внешний HTTP
  мокируется (`FakeIPS` / локальный aiohttp-сервер), без реальной сети в unit-тестах.
- На каждый багфикс — регрессионный тест.

### E2E на проде — обязательно

Юнит-теста на fake-сервере **недостаточно**: он проверяет код против предполагаемых ключей JSON, а не
против реального ответа IPS. Поэтому **каждый метод перед статусом «готов» проверяется на боевом
сервере** (креды из окружения `IPS_*`, не в репозитории), а факт фиксируется во внутреннем реестре
прод-проверок (дата + что проверено + результат). Это ловит молчаливую потерю
данных (например, регистр алиасов) и расхождения схем со swagger. Разрушающие операции на проде
выполняются только через одноразовый объект с обязательным удалением.

## Git и безопасность

- Репозиторий приватный (SSH). **Conventional Commits**: `feat:`, `fix:`, `chore:`, `test:`,
  `refactor:`, `docs:`, `perf:`. Коммит на логический блок, пуш каждые 1–3 коммита. **Никогда не
  amend** — только новые коммиты.
- Никогда не коммитить секреты/PII: реальный адрес сервера IPS, логины, пароли существуют только в
  gitignored `.env` или в окружении — **никогда** в коде, тестах, фикстурах, примерах, документации.
  В репозитории — плейсхолдеры (`your-ips-host`, `OWNER`) и `.env.example` с пустыми значениями.
- Pre-commit (gitleaks + detect-private-key) отвергает staged-секреты. Конфиг — через
  `pydantic-settings` (env-prefix `IPS_`), без хардкода кред/URL/путей.

## Как добавить новый метод (чек-лист)

1. Определить раздел и имя по [конвенции имён](#adr-0003) из адреса эндпоинта.
2. Создать `methods/<раздел>/<метод>.py` — mixin от `APIManager`, вызывающий `self._request(...)`;
   развернуть result-обёртку при необходимости.
3. Создать парную схему в `schemas/<раздел>/<метод>.py` (pydantic v2, camelCase-алиасы).
4. Написать docstring уровня MCP (см. выше).
5. Зарегистрировать mixin в `__init__` раздела и (если новый раздел) в `client.py`.
6. Добавить осмысленный тест на fake-сервере; на разрушающий метод — проверить гейт `confirm`.
7. Прогнать гейт `ruff + mypy + pytest` локально до зелёного.
8. Проверить метод на проде и записать факт во внутренний реестр прод-проверок.

Подробности архитектуры — в разделе [Архитектура](architecture.md).
