# Конфигурация

Прежде чем сделать запрос, клиенту нужно знать **куда** подключаться и **под кем**. За это отвечает
класс `IPSConfig`. На этой странице — все его параметры, как задать их через переменные окружения
`IPS_*` и файл `.env`, и какой минимум обязателен.

## Что такое `IPSConfig`

`IPSConfig` — это pydantic-settings модель: один объект, который хранит адрес сервера,
авторизационные данные и тонкие настройки транспорта (таймауты, повторы, TLS). Вы передаёте его в
`IPSClient`, и клиент берёт оттуда всё необходимое.

Значения подхватываются в порядке приоритета:

1. **явные аргументы конструктора** `IPSConfig(...)`;
2. **переменные окружения** с префиксом `IPS_`;
3. **файл `.env`** в корне проекта.

Имя переменной окружения = `IPS_` + имя поля в верхнем регистре (регистр не важен). Например,
поле `base_url` читается из `IPS_BASE_URL`, `login_name` — из `IPS_LOGIN_NAME`. Неизвестные
переменные игнорируются.

```python
from aioips import IPSConfig

# Явно
config = IPSConfig(
    base_url="http://your-ips-host:8080",
    login_name="your-login",
    password="...",
    role_name="Администратор",
)

# Или целиком из окружения / .env — без аргументов
config = IPSConfig()
```

## Параметры

### Подключение и авторизация

| Поле | Переменная окружения | Тип | По умолчанию | Назначение |
|---|---|---|---|---|
| `base_url` | `IPS_BASE_URL` | `str` | **обязателен** | URL сервера IPS Web API, например `http://your-ips-host:8080`. Должен начинаться с `http://` или `https://` |
| `login_name` | `IPS_LOGIN_NAME` | `str \| None` | `None` | Имя пользователя IPS для входа |
| `password` | `IPS_PASSWORD` | `str \| None` | `None` | Пароль пользователя IPS |
| `password_type` | `IPS_PASSWORD_TYPE` | `"plainText" \| "base64Text"` | `"plainText"` | Способ передачи пароля |
| `role_name` | `IPS_ROLE_NAME` | `str \| None` | `None` | Имя роли; автоматически резолвится в идентификатор, если `role_id` не задан |
| `role_id` | `IPS_ROLE_ID` | `int \| None` | `None` | Числовой идентификатор роли (если известен заранее) |
| `access_level_id` | `IPS_ACCESS_LEVEL_ID` | `int` | `0` | Идентификатор уровня доступа (`0` — обычный) |
| `access_token` | `IPS_ACCESS_TOKEN` | `str \| None` | `None` | Заранее полученный JWT access-токен (альтернатива логину/паролю) |

### Транспорт, повторы и логирование

| Поле | Переменная окружения | Тип | По умолчанию | Назначение |
|---|---|---|---|---|
| `request_timeout` | `IPS_REQUEST_TIMEOUT` | `float` | `30.0` | Таймаут одного запроса, секунды |
| `max_retries` | `IPS_MAX_RETRIES` | `int` | `3` | Число повторов при транзиентных ошибках (0–10) |
| `retry_min_wait` | `IPS_RETRY_MIN_WAIT` | `float` | `1.0` | Минимальная задержка между повторами, секунды |
| `retry_max_wait` | `IPS_RETRY_MAX_WAIT` | `float` | `10.0` | Максимальная задержка между повторами, секунды (не меньше `retry_min_wait`) |
| `connector_limit` | `IPS_CONNECTOR_LIMIT` | `int` | `100` | Лимит одновременных HTTP-соединений |
| `verify_ssl` | `IPS_VERIFY_SSL` | `bool` | `True` | Проверять ли TLS-сертификат сервера |
| `token_refresh_skew` | `IPS_TOKEN_REFRESH_SKEW` | `float` | `30.0` | Запас в секундах: токен обновляется заранее, до истечения |
| `log_level` | `IPS_LOG_LEVEL` | `str` | `"ERROR"` | Уровень логирования: `DEBUG`/`INFO`/`WARNING`/`ERROR`/`CRITICAL` |
| `log_json` | `IPS_LOG_JSON` | `bool` | `False` | Выводить логи в структурном JSON-формате |

## Два способа авторизации

`IPSConfig` требует **ровно один** валидный набор авторизационных данных, иначе при создании
конфигурации будет `ValidationError`:

1. **Логин и пароль** — `login_name` + `password` (плюс по желанию `role_name`/`role_id` и
   `access_level_id`). Клиент сам получит JWT-токен и будет обновлять его автоматически.
2. **Готовый токен** — `access_token`, полученный заранее. В этом случае логин/пароль не нужны,
   но клиент не сможет перевыпустить истёкший токен.

Подробно различия разобраны на странице [Авторизация](authentication.md).

!!! warning "Минимально необходимое"
    Обязателен только `base_url`. Кроме него нужен **один** из наборов: либо
    `login_name` + `password`, либо `access_token`. Если не задать ни одного — конфигурация не
    создастся.

## Файл `.env`

Самый удобный способ не держать секреты в коде — положить их в `.env` в корне проекта. Клиент
прочитает его автоматически.

```dotenv
# .env — НЕ коммитить (он в .gitignore)
IPS_BASE_URL=http://your-ips-host:8080
IPS_LOGIN_NAME=your-login
IPS_PASSWORD=...
IPS_ROLE_NAME=Администратор
```

После этого конфигурацию можно создать без единого аргумента:

```python
from aioips import IPSConfig

config = IPSConfig()  # всё подхватится из .env / окружения
```

!!! danger "Безопасность"
    Реальные адреса, логины, пароли и токены существуют **только** в `.env` или в переменных
    окружения — никогда в коде, тестах, фикстурах или примерах. В репозитории есть
    `.env.example` с пустыми значениями как шаблон; копируйте его в `.env` и заполняйте локально.
