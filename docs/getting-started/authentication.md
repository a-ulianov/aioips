# Авторизация

IPS Web API работает по JWT: чтобы выполнить запрос, нужен короткоживущий токен, который к тому же
надо вовремя обновлять. Хорошая новость — `aioips` делает это за вас. На этой странице простыми
словами: какие есть способы входа, как клиент сам получает и обновляет токен, и как разобраться
с ролями и уровнями доступа.

## Главное в двух словах

Вы задаёте авторизационные данные в [`IPSConfig`](configuration.md) — и забываете про токены.
Клиент сам войдёт при первом запросе, сам обновит токен перед истечением и сам повторит запрос,
если сервер вдруг ответил «токен недействителен». Никаких ручных вызовов «login» в вашем коде нет.

## Два способа войти

`aioips` поддерживает два взаимоисключающих способа авторизации. Нужно задать **ровно один**.

### Способ 1. Логин и пароль (рекомендуемый)

Вы передаёте `login_name` и `password` — клиент сам получает JWT и автоматически его обновляет
и перевыпускает. Это основной режим: токен живёт «сам по себе», вам ничего не нужно делать.

```python
from aioips import IPSConfig

config = IPSConfig(
    base_url="http://your-ips-host:8080",
    login_name="your-login",
    password="...",          # лучше — через переменную окружения IPS_PASSWORD
    role_name="Администратор",
)
```

### Способ 2. Готовый access-токен

Если токен уже получен где-то ещё, передайте его в `access_token` — без логина и пароля:

```python
config = IPSConfig(
    base_url="http://your-ips-host:8080",
    access_token="...",
)
```

!!! warning "У готового токена нет «второго дыхания»"
    Без логина/пароля клиент не сможет перевыпустить истёкший токен — при истечении вы получите
    `IPSAuthError`. Способ с токеном удобен для коротких задач; для долгоживущих процессов берите
    логин/пароль.

## Как это работает под капотом

IPS использует двухшаговую схему авторизации, и клиент полностью её скрывает:

1. **Опции входа** — `GET /core/api/Auth/logins/{loginName}/options`: какие роли и уровни доступа
   доступны логину. Пароль не нужен. Этот шаг клиент выполняет сам, когда нужно превратить имя
   роли в её числовой идентификатор.
2. **Аутентификация** — `POST /core/api/Auth/authenticate`: обмен логина/пароля на пару токенов
   `{accessToken, refreshToken, expireTime}`.
3. **Обновление** — `POST /core/api/Auth/refreshTokens`: обмен текущей пары токенов на новую.

Стратегия управления токеном (см. [ADR-0002](../architecture.md)):

| Когда | Что делает клиент |
|---|---|
| **Ленивый вход** | Токен не запрашивается заранее — только при первом запросе, которому нужна авторизация |
| **Проактивное обновление** | Если до истечения токена осталось меньше `token_refresh_skew` секунд (по умолчанию 30) — обновляет заранее, до того как запрос упадёт |
| **Реактивный повтор по `401`** | Если сервер всё же ответил `401`, клиент один раз обновляет токен и повторяет запрос |
| **Откат к полному входу** | Если обновление не удалось — выполняет повторную аутентификацию по логину/паролю (если они заданы) |

!!! note "Зачем и проактивно, и реактивно"
    Проактивное обновление по `expireTime` экономит лишний запрос, а реактивный повтор по `401`
    страхует от рассинхрона часов клиента и сервера. Вместе они дают надёжность без вашего участия.

Параллельные корутины не приводят к многократному входу — доступ к токену защищён внутренней
блокировкой.

## Роли и уровни доступа

Один пользователь может входить под разными **ролями** и **уровнями доступа** — от этого зависит,
что он увидит и сможет сделать. Выбор задаётся в конфигурации:

- `role_name` — имя роли. Клиент сам резолвит его в числовой `role_id` через «опции входа».
- `role_id` — если идентификатор роли известен заранее, можно задать его напрямую (тогда лишнего
  запроса опций не будет).
- `access_level_id` — уровень доступа (`0` — обычный).

Если роль не задана вовсе, сервер выберет роль по умолчанию.

### Посмотреть доступные роли заранее

Метод `login_options` показывает, под какими ролями и уровнями доступа логин может войти — **до**
аутентификации, пароль не требуется. Удобно, чтобы узнать корректное имя или `id` роли.

```python
async with IPSClient(config=config) as ips:
    options = await ips.login_options("your-login")

    # Роли: сопоставление имя -> roleID
    for role in options.roles:
        print(role.id, role.name)

    # Уровни доступа: accessLevelID -> имя
    for level in options.access_levels:
        print(level.id, level.name)
```

`login_options` возвращает объект `AuthOptions` с двумя списками: `roles` (у каждой роли `id` и
`name`) и `access_levels` (у каждого уровня `id` и `name`). Значение `role.id` — это тот самый
`role_id`, который можно подставить в конфигурацию.

!!! tip "Ошибка про недоступную роль"
    Если указать `role_name`, которого нет у пользователя, при входе будет `IPSAuthError` со
    списком доступных ролей. Сверьте имя через `login_options` — оно должно совпадать точно.

## Проверить, кто вошёл

После входа метод `user_info()` показывает, под какой ролью и уровнем фактически работает сессия
и есть ли админ-права. Это надёжный способ убедиться, что авторизация удалась.

```python
async with IPSClient(config=config) as ips:
    me = await ips.user_info()
    print(me.login_name)   # имя для входа
    print(me.is_admin)     # есть ли административные права
    print(me.access_level) # идентификатор уровня доступа сессии
```

!!! note "`login_options` против `user_info`"
    `login_options` показывает варианты входа **до** аутентификации (что *доступно*), а
    `user_info` — что *фактически* активно **после** входа.

## Если что-то пошло не так

- **`IPSAuthError`** — неверный логин/пароль, недоступная роль, либо истёк готовый токен, который
  нечем перевыпустить. Проверьте креды в [`.env`](configuration.md) и имя роли через
  `login_options`.
- **Повторяющиеся `401`** — обычно рассинхрон часов или отозванные права. Клиент сам делает один
  повтор с обновлением; если и он не помогает, проблема на стороне сервера/учётной записи.
