# Справочник API

**Что это.** Полный перечень методов клиента `aioips` — асинхронной обёртки над IPS Server Web API.
Каждый метод клиента соответствует одной операции сервера: вы вызываете `await ips.<метод>(...)`,
получаете готовую pydantic-схему. Справочник сгруппирован по разделам (как в самом IPS) и нужен,
**когда вы уже знаете, что хотите сделать, и ищете нужный метод и его сигнатуру**.

**Когда сюда заглядывать.** За концепциями (что такое объект, версия, состав, жизненный цикл) идите
в [Концепции](../concepts/index.md); за установкой и первым вызовом — в
[Начало работы](../getting-started/index.md). Справочник — это «телефонный справочник»: краткое
назначение метода и его параметры, без вводных пояснений.

## Как устроен справочник

Все методы клиента собраны множественным наследованием в единый класс `IPSClient`, поэтому
**имя метода глобально уникально** и уже несёт контекст раздела (`object_*`, `attribute_*`,
`relation_*`, `life_cycle_*`). Имена нормализованы по [соглашению об именовании](../contributing.md):
берётся часть адреса эндпоинта после `/api/`, сегменты-параметры отбрасываются.

Каждая страница раздела устроена одинаково:

- вверху — 2–3 предложения «что за раздел и когда он нужен»;
- далее — компактные таблицы `Метод | Назначение` с подзаголовками-группами;
- разрушающие (необратимые) методы помечены `confirm=True` — без явного `confirm=True` они бросают
  ошибку, не выполняя действие.

## Общие принципы (читать один раз)

### Два пространства идентификаторов (важно!)

В IPS объект и его версия — разные сущности с разными id. Путаница здесь — частая причина «объект не
найден». Подробно — в [Модели данных](../concepts/data-model.md), кратко:

| Поле в DTO | Что адресует | Где принимается |
|---|---|---|
| `objectID` (camel: `objectID`) | **ОБЪЕКТ** (общий для всех версий) | `object_get`, `object_info`, `objects_collection` |
| `id` | **ВЕРСИЯ** объекта | методы версий, `object_security`, обсуждения |
| `objectGUID` | ОБЪЕКТ (глобально) | `object_get_by_guid`, `object_info_by_guid` |
| `guid` | ВЕРСИЯ (глобально) | `*_by_guid` версий |
| `projID` / `partID` | в связи: `projID` = id ОБЪЕКТА-родителя, `partID` = id ВЕРSИИ-потомка | методы `relations` |

Правило: по id версии объект через `object_get` **не достаётся** — нужен id объекта.

### Разворот result-обёрток `*NullableResultDto` → `Schema | None`

IPS оборачивает «может не быть результата» в DTO вида `{ entity, isEntityPresent }`. Клиент
разворачивает такую обёртку **внутри метода**: наружу возвращается либо сама схема, либо `None`
(когда `isEntityPresent = false`). Деталь транспорта не протекает в ваш код — просто проверяйте
результат на `None`.

### Confirm-гейты на разрушающих операциях

Необратимые и массовые операции (удаление объекта/связи/снимка, очистка атрибута, отмена правок,
исключение из состава, удаление истории) принимают параметр `confirm: bool = False`. Без
`confirm=True` метод бросает ошибку и **ничего не делает** — это защита от случайного запуска.
Полный список таких методов помечен в таблицах меткой `confirm=True`.

### Чтение vs запись и жизненный цикл

Методы чтения идемпотентны и безопасны. Методы записи атрибутов работают **только** в верном режиме
правки: цикл `object_check_out` → правка → `object_check_in` (или `object_cancel_changes`). Создание
объекта не завершено без `object_commit_creation`. См. [Жизненный цикл](../concepts/lifecycle.md).
