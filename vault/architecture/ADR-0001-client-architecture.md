# ADR-0001. Архитектура клиента: mixin-разделы поверх общего ядра

- Дата: 2026-06-24
- Статус: принято

## Контекст
`aioips` — обёртка над IPS Server Web API (744 операции, см. [[ips-web-api-overview]]).
Нужна структура, которая масштабируется на сотни эндпоинтов и остаётся читаемой.

## Решение
Применяем проверенный паттерн: **один метод API = один файл = один mixin-класс**, плюс парный
файл схемы и парный тест. Разделы (`auth`, `users`, `metadata`, …) собираются множественным
наследованием в единый класс `IPSClient`. Общее поведение (HTTP, авторизация, повторы,
ошибки) живёт в `core/` и наследуется всеми mixin'ами через базовый `APIManager`.

Слои и поток запроса описаны в [[layers]].

## Сознательные отклонения от типового паттерна
- **Одна сессия на экземпляр клиента** вместо общего пула соединений по `client_id`. Общий пул
  оправдан, когда per-account лимиты соблюдаются при статичном ключе; у IPS авторизация — это
  per-session JWT, поэтому общий пул не даёт выгоды и усложняет код. См. [[ADR-0002-auth-token-lifecycle]].
- **Нет per-method rate-limiter.** У IPS нет документированных жёстких лимитов; добавим
  позже при необходимости, не усложняя ядро преждевременно (KISS).
- **Алиасы `snake_case` ↔ `camelCase`** через `pydantic.alias_generators.to_camel`
  (поля IPS Web API — `camelCase`, поэтому нужен генератор алиасов).

## Последствия
- Добавление метода = 3 маленьких файла (метод, схема, тест) + строка в `__init__` раздела.
- Плоская, предсказуемая структура; легко делегировать генерацию разделов.
- `IPSClient` — «толстый» по числу методов, но каждый метод изолирован.
