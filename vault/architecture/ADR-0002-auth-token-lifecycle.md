# ADR-0002. Жизненный цикл JWT-токена

- Дата: 2026-06-24
- Статус: принято

## Контекст
IPS использует короткоживущий JWT: `POST /authenticate` возвращает
`{accessToken, refreshToken, expireTime}`, токен надо обновлять. Это отличается от API со
статичным ключом. Подробности эндпоинтов — в [[auth]].

## Решение
Выделен `core/auth.py::AuthManager`, инкапсулирующий состояние токена. Стратегия:

1. **Lazy-auth.** Токен не запрашивается до первого вызова, требующего авторизации.
   `ensure_access_token()` аутентифицируется при отсутствии токена.
2. **Проактивное обновление по истечению.** Перед запросом, если до `expireTime` осталось
   меньше `token_refresh_skew` секунд, выполняется `refresh`.
3. **Реактивное обновление по 401.** Если сервер вернул `401`, ядро (`core.py::_request`)
   один раз вызывает `force_refresh()` и повторяет запрос.
4. **Fallback.** Если `refresh` не удался — повторная полная аутентификация (если есть
   логин/пароль). Если есть только готовый токен без логина/пароля — `IPSAuthError`.
5. **Конкурентность.** Доступ к токену защищён `asyncio.Lock`, чтобы параллельные корутины
   не аутентифицировались многократно.

`AuthManager` получает низкоуровневую функцию запроса (`_raw_request`) инъекцией, не завязан
на ядро напрямую — это упрощает тестирование (см. `FakeTransport` в тестах auth).

## Последствия
- Заголовок `Authorization` добавляется per-request (токен меняется), не вшит в сессию.
- Резолвинг роли по имени (`role_name` → `roleID`) идёт через `login options` при первой
  аутентификации.
- Тонкое место: рассинхрон часов клиента и сервера. Смягчается реактивным 401-retry в
  дополнение к проактивному обновлению.
