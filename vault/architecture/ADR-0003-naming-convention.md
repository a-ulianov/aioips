# ADR-0003. Соглашение об именовании методов и схем

- Дата: 2026-06-24
- Статус: принято

## Контекст
Имена методов, файлов и схем нужно выводить детерминированно из адресов эндпоинтов, чтобы
обёртка масштабировалась на 744 операции без споров об именах. Требование заказчика: «имена
методов и схем берём из адресов методов — всё, что после `/api/`».

## Решение
Для каждого эндпоинта берём часть пути **после `/api/`** (работает и для `/core/api/...`, и
для `/api/...`), отбрасываем сегменты-параметры `{...}`.

1. **Каталог (mixin-раздел)** = первый сегмент после `/api/`. Пример: `objects`, `metadata`,
   `files`. Это директория в `methods/<раздел>/` и `schemas/<раздел>/`.
2. **Имя файла/метода** = оставшиеся непараметрические сегменты в `snake_case`, соединённые `_`.
   - `…/objects/CreateByPrototype` → `create_by_prototype`
   - `…/objects/{objectId}/objectInfo` → `object_info`
   - `…/objects/byGuid/{guid}/objectInfo` → `object_info_by_guid`
3. **CRUD на коллекции/`{id}` без описательного хвоста** — добавляем глагол по HTTP-методу,
   опираясь на `operationId` (он на английском и точен): `GET /objects/{id}` →
   `object_get` (Objects_GetObject); `POST /objects` → `object_create` (Objects_Create).
4. **Схемы** — `PascalCase` того же имени + суффикс роли: `…Request`, `…Response`, `…Item`.
   Имена файлов схем совпадают с именами файлов методов.
5. **Имя метода клиента** глобально уникально (все mixin'ы плоско в `IPSClient`), поэтому
   несёт контекст раздела (имя уже начинается с сущности раздела: `object_*`, `attribute_*`).

`operationId` IPS имеет вид `Category_Action` — используем его как ориентир при неоднозначном
пути и как основу для англоязычного резюме в docstring.

## Источник docstrings
`summary`/`description` в OpenAPI-спеке IPS повреждены (кириллица → U+FFFD, см. [[gotchas]]),
поэтому назначение метода описываем по: (1) `operationId`, (2) семантике пути и параметров,
(3) именам полей схем, (4) Руководству администратора IPS (терминология: версия объекта,
шаг ЖЦ, checkout/checkin, состав, GUID версии vs GUID объекта).

## Последствия
- Имена предсказуемы и выводимы из адреса; генерацию разделов можно делегировать субагентам.
- Result-обёртки IPS (`...NullableResultDto {entity, isEntityPresent}`) разворачиваются в
  методе: наружу отдаём `Schema | None`, не протекая деталь транспорта.
