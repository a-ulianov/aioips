# Базовые замеры производительности (2026-06-26)

Замеры на боевом IPS (`IPS_BASE_URL`, LAN), Python 3.13, дефолтная конфигурация
(`connector_limit=100`, `max_retries=3`). Воспроизводить: `IPSClient` + `time.perf_counter`.

| Метрика | Значение | Заметка |
|---|---|---|
| Латентность лёгкого чтения (`user_info`) | avg **3.7 мс**, median 3.5, min 3.1 | упирается в сетевой RTT LAN; накладные обёртки пренебрежимы |
| Латентность тяжёлого (`object_types`, 988 типов) | avg **63 мс**, median 60, p95 73, max 88 | большой ответ + pydantic-валидация |
| Throughput, 10 конкур. `user_info` | **~966 req/sec** | |
| Throughput, 50 конкур. `user_info` | **~1 950 req/sec** (пик) | ограничено `connector_limit` и сервером |
| Большой payload (`find_collection`, 50 800 объектов) | 5.17 с → **~9 800 объ/сек**, **~18.3 МБ/с** | включая pydantic-валидацию 50.8k DTO |

**Выводы:**
- Узкое место чтений — сетевой RTT (LAN ~3-4 мс), не код обёртки.
- Async-конкуренция масштабируется до ~2000 req/sec; для большего поднять `connector_limit`.
- Узкое место больших выборок — **десериализация/валидация pydantic**, не транспорт. Для 50k+
  объектов рассмотреть: отключение валидации (`model_construct`), стриминг, keyset-пагинацию.
- `find_collection`/`objects_select` на типах с десятками тысяч объектов (1742 → 50 800) грузят всё
  в память — для production добавить пагинацию-хелпер.
